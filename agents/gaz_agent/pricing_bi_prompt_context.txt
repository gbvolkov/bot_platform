This database contains normalized model-level comparison data extracted from selected GAZ Excel sheets.

Main working table:
- comparisons_normalized

There are no option, trim, service-schedule, or raw-cell tables in this database.
- If the user asks about options, trims, maintenance schedule, or service costs, do not invent SQL against missing tables.
- Answer only from comparisons_normalized, or clearly state that the current database does not contain that layer.

String matching convention:
- Every user-facing text field has a mirror field with the `_nocase` suffix.
- `_nocase` stores the same value lowercased.
- Use the original field in SELECT output.
- Use the `_nocase` field in WHERE, LIKE, joins, and candidate lookup.

Key interpretation rules:
- `id` is the primary normalized model identifier. It is already lowercase and ASCII-normalized.
- `comp_brand` and `comp_model` are the two canonical identity parts used to build `id`.
- `comp_full_name` is a display/helper field derived from `comp_brand + comp_model`.
- Do not assume `comp_full_name` is the only identity field.
- For exact model aggregation, prefer grouping by `id`.
- For candidate search from a user query, use `comp_full_name_nocase`, `comp_brand_nocase`, and `comp_model_nocase`.
- `base_model` is the canonical base-model field.
- `comp_brand` mirrors `base_model`.
- `base_model_nocase` and `comp_brand_nocase` are equal lowercase mirrors of the same value.

General SQL strategy:
- If the user asks about a specific model or modification, first locate candidate rows by `comp_full_name_nocase`, `comp_brand_nocase`, or `comp_model_nocase`, then aggregate by `id`.
- If the user asks about a model family in a generic way, aggregate across all matching `id` values.
- Do not use `LIMIT 1` for model answers without aggregation.
- Null in one row does not mean the attribute is unavailable globally; different source sheets may have filled different fields before merge.

==================================================
TABLE: comparisons_normalized
==================================================

Purpose:
- One merged row per normalized model id.
- Stores prices, core technical specifications, warranty, service interval, and a small amount of ownership-cost data.

Important fields:
- `id`
  Primary normalized identifier for the merged model row.

- `source_file`, `source_sheet`
  Provenance of the first non-empty source row that won the merge.
  Use only when the user explicitly asks about origin.

- `base_model`
  Canonical base-model field. Do not treat it as workbook context.

- `comp_brand`
  Brand/display alias that mirrors `base_model`.

- `comp_model`
  Second canonical identity part.

- `comp_full_name`
  Human-readable combined model name derived from canonical parts.

- `price_rub_min`, `price_rub_max`, `price_comment`
  Price range and price note.

- `vehicle_type`, `body_type`
  Structured vehicle category fields.
  `body_type` uses normalized English values such as `flatbed`, `panel_van`, `chassis`, `bus`, `minivan`.

- `passenger_capacity`, `seat_count`, `cab_seat_count`
  Seating and passenger capacity fields.

- `engine_fuel_type`, `engine_power_hp`, `engine_power_kw`, `engine_volume_l`
  Engine characteristics.

- `transmission_type`, `transmission_gears`, `drive_type`, `wheel_formula`
  Drivetrain and transmission characteristics.

- `gross_weight_kg`, `curb_weight_kg`, `payload_kg`
  Weight metrics.

- `length_mm`, `width_mm`, `height_mm`, `wheelbase_mm`, `ground_clearance_mm`
  Dimensions.

- `fuel_tank_l`, `fuel_consumption_l100km`
  Fuel metrics.

- `service_interval_km`, `service_interval_months`
  Service interval data when present in source.

- `ownership_cost_rub_km`
  Ownership cost per km when present.

- `warranty_months`, `warranty_km`
  Warranty period and mileage.

- `notes`
  Residual technical notes that were not mapped to structured fields.

Practical SQL rules:
- For exact model answers, aggregate by `id`.
- For fuzzy user input, first find candidates with `LIKE` on `comp_full_name_nocase`, `comp_brand_nocase`, and `comp_model_nocase`.
- If the user asks for prices, prefer `MIN(price_rub_min)` and `MAX(price_rub_max)` across the matched ids.
- If the user asks for technical specs, select the requested structured fields from the matched ids and ignore rows where those fields are null.
- `base_model` is canonical for filtering; `comp_brand` is an equivalent mirror.
- Ignore provenance/order fields in normal business answers unless the user asks about data source.

Anti-patterns:
- Do not query missing tables such as `comparison_model_options`, `comparison_service_groups`, or `comparisons_raw_params`.
- Do not assume `base_model` identifies the requested model by itself.
- Do not use case-sensitive matching when `_nocase` exists.
- Do not search `body_type_nocase` with Russian substrings; map business words to normalized English values first.
