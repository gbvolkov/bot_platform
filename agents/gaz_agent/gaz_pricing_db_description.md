# База данных `gaz_pricing.sqlite`

Файл базы данных: `C:\Projects\bot_platform\data\gaz-pricing\gaz_pricing.sqlite`

База содержит нормализованные данные из Excel-таблиц сравнения моделей GAZ с конкурентами. Основные рабочие таблицы:

- `comparisons_normalized` — модельный уровень: цены и технические параметры.
- `comparison_model_options` — комплектации, опции, статусы опций и цены допопций.
- `comparison_service_groups` — данные по ТО и сервисным интервалам.

Таблица `comparisons_raw_params` здесь не описывается.

## Общие правила интерпретации данных

1. `base_model` — это не "базовая модель" и не "базовая комплектация". Это модель GAZ, относительно которой построено сравнение на конкретном листе Excel.
2. Одна и та же модель может встречаться в нескольких строках, листах и файлах. Поэтому нельзя считать одну строку канонической записью по модели.
3. Если вопрос задан по модели в общем виде, без точного указания модификации, ответ нужно строить по всем найденным модификациям этой модели, а не по одной произвольно выбранной версии.
4. Для получения ответа по модели нужно всегда собирать обобщённую информацию по нескольким записям этой модели.
5. В отдельных строках цена может отсутствовать. Если `price_rub_min` и `price_rub_max` пусты, цену надо искать в других записях той же модели.
6. Для каждого строкового поля в рабочей схеме существует зеркальное поле с суффиксом `_nocase`, содержащее то же значение в нижнем регистре.
7. Поиск и фильтрация по строковым полям должны выполняться через поля `_nocase`, а для вывода нужно использовать исходные поля без этого суффикса.
8. Для идентификации модели в первую очередь используется `comp_full_name`. `comp_brand` — вспомогательное поле для группировки и фильтрации по бренду.
9. При ответах по опциям и комплектациям нужно использовать `comparison_model_options`, а не пытаться выводить их из `comparisons_normalized`.
10. При ответах по ТО и сервисным затратам нужно использовать `comparison_service_groups`.

## Таблица `comparisons_normalized`

Таблица содержит нормализованные записи по моделям: диапазоны цен, базовые технические характеристики, гарантию, межсервисные интервалы и отдельные показатели владения.

### Поля

- `source_file`
  Исходный Excel-файл, из которого получена запись.

- `source_sheet`
  Исходный лист Excel, из которого получена запись.

- `sheet_type`
  Внутренний тип листа после разбора.
  Тип используется для понимания происхождения записи, например технический лист, краткий лист, лист по стоимости владения.

- `base_model`
  Модель GAZ, по отношению к которой построено сравнение на листе.
  Это ключевой контекст сравнения, а не название базовой версии модели.

- `comp_full_name`
  Полное наименование модели из Excel.
  Это основное поле для сопоставления записей одной и той же модели между разными листами.

- `comp_brand`
  Бренд модели из Excel.
  Используется как вспомогательное поле для фильтрации и группировки, но не заменяет `comp_full_name`.

- `price_rub_min`
  Минимальная найденная цена модели в рублях.
  Если в исходной ячейке была одна цена, минимальное и максимальное значения совпадают.
  Если поле пустое, это не означает, что цена модели неизвестна во всей базе: её нужно искать в других строках той же модели.

- `price_rub_max`
  Максимальная найденная цена модели в рублях.
  Используется вместе с `price_rub_min` как ценовой диапазон.
  Если поле пустое, нужно искать цену в других записях по той же модели.

- `price_comment`
  Текстовое пояснение по цене.
  Сюда попадают примечания из Excel, если цена в листе сопровождалась дополнительным комментарием.

- `vehicle_type`
  Тип транспортного средства.
  Примеры: автобус, минивэн, бортовой автомобиль и другие типы, как они были отражены в листах сравнения.

- `body_type`
  Тип кузова или исполнения.
  Значение берётся из исходных сравнительных листов, если соответствующая строка там была.

- `passenger_capacity`
  Пассажировместимость.
  Обычно это общее число пассажиров, если такое поле было указано в Excel.

- `seat_count`
  Количество мест.
  Используется для значений, относящихся именно к числу посадочных мест.

- `cab_seat_count`
  Количество мест в кабине.
  Применяется там, где Excel отдельно выделял места в кабине.

- `engine_fuel_type`
  Тип топлива двигателя.
  Например: дизель, бензин, газ или другие значения, явно присутствующие в исходных листах.

- `engine_power_hp`
  Мощность двигателя в лошадиных силах.
  Заполняется только когда значение удалось однозначно извлечь из листа.

- `engine_power_kw`
  Мощность двигателя в киловаттах.
  Заполняется только когда значение удалось однозначно извлечь из листа.

- `engine_volume_l`
  Рабочий объём двигателя в литрах.

- `transmission_type`
  Тип коробки передач в том виде, в котором он представлен в Excel.
  Здесь хранится исходный смысловой тип, например `МКПП`, `АКПП`, `6MT` и подобные значения.

- `transmission_gears`
  Количество передач.
  Если число передач не было указано явно или не извлекалось однозначно, поле остаётся пустым.

- `drive_type`
  Тип привода.
  Например: задний, полный, передний.

- `wheel_formula`
  Колёсная формула.
  Например: `4x2`, `4x4`.

- `gross_weight_kg`
  Нижняя граница полной массы автомобиля в килограммах. Если в источнике было одно число, оно совпадает с `gross_weight_kg_min` и `gross_weight_kg_max`.

- `gross_weight_kg_min`, `gross_weight_kg_max`
  Нижняя и верхняя границы полной массы автомобиля в килограммах.

- `curb_weight_kg`
  Нижняя граница снаряжённой массы автомобиля в килограммах. Если в источнике было одно число, оно совпадает с `curb_weight_kg_min` и `curb_weight_kg_max`.

- `curb_weight_kg_min`, `curb_weight_kg_max`
  Нижняя и верхняя границы снаряжённой массы автомобиля в килограммах.

- `payload_kg`
  Нижняя граница грузоподъёмности в килограммах. Если в источнике было одно число, оно совпадает с `payload_kg_min` и `payload_kg_max`.

- `payload_kg_min`, `payload_kg_max`
  Нижняя и верхняя границы грузоподъёмности в килограммах.

- `length_mm`
  Длина в миллиметрах.

- `width_mm`
  Ширина в миллиметрах.

- `height_mm`
  Высота в миллиметрах.

- `wheelbase_mm`
  Колёсная база в миллиметрах.

- `ground_clearance_mm`
  Дорожный просвет в миллиметрах.

- `fuel_tank_l`
  Объём топливного бака в литрах.

- `fuel_consumption_l100km`
  Расход топлива в литрах на 100 км.

- `service_interval_km`
  Межсервисный интервал в километрах.

- `service_interval_months`
  Межсервисный интервал в месяцах.

- `ownership_cost_rub_km`
  Стоимость владения или эксплуатационные затраты в рублях на 1 км.

- `warranty_months`
  Срок гарантии в месяцах.

- `warranty_km`
  Ограничение гарантии по пробегу в километрах.
  Если в Excel было указано значение вроде "без ограничения", числовое поле может быть пустым, а уточнение будет отражено в `notes`.

- `notes`
  Технические примечания, которые не удалось корректно разложить по отдельным числовым полям.
  Это дополнительное поясняющее поле, а не основной источник структурированных данных.

### Как использовать таблицу `comparisons_normalized`

- Для поиска цены по модели нельзя ограничиваться одной строкой. Надо собирать все записи по одному `comp_full_name` и смотреть все непустые `price_rub_min` и `price_rub_max`.
- Если пользователь спрашивает модель без точной модификации, нужно собирать диапазон цены и другие сведения по всем найденным модификациям этой модели.
- Поиск по строковым полям, таким как `comp_full_name`, `comp_brand`, `option_name`, `group_name` и аналогичным, нужно выполнять через соответствующие поля `_nocase`.
- Для ответа по модели полезно агрегировать записи по `comp_full_name`, а при необходимости дополнительно учитывать `comp_brand`.
- Если по одной строке часть параметров пустая, это не означает, что параметр отсутствует по модели в целом. Он может быть заполнен в другой записи той же модели.
- Один и тот же `comp_full_name` может встречаться под разными `base_model`, потому что одна и та же конкурентная модель сравнивается с разными моделями GAZ.

## Таблица `comparison_model_options`

Таблица содержит комплектации, опции, статусы наличия опций и цены допопций.

### Поля

- `source_file`
  Исходный Excel-файл, из которого получена запись по опции.

- `source_sheet`
  Исходный лист Excel, из которого получена запись по опции.

- `base_model`
  Модель GAZ, относительно которой сформирован этот сравнительный лист.
  Это контекст сравнения, а не название базовой версии модели.

- `comp_full_name`
  Полное наименование модели, к которой относится опция.

- `comp_brand`
  Бренд модели, к которой относится опция.

- `option_group`
  Группа опций или раздел листа, если в Excel опции были сгруппированы по разделам.

- `option_name`
  Наименование опции в точной или максимально близкой форме к Excel.
  Это основное поле для поиска конкретной опции.

- `option_status_raw`
  Исходный текстовый статус опции из Excel.
  Например: `база`, `опция`, `+`, `-`, `недоступен` и подобные значения.

- `option_status_norm`
  Нормализованный статус опции.
  Используемые значения:
  `base` — опция входит в базовую комплектацию.
  `included` — опция включена.
  `optional` — опция доступна как дополнительная.
  `not_present` — опция отсутствует.
  `unavailable` — опция недоступна.
  `unknown` — статус неоднозначен или непонятен.
  `listed` — опция перечислена на листе без явного статуса.

- `option_price_raw`
  Исходное текстовое значение цены опции из Excel.
  Может содержать как саму цену, так и смешанный текст вроде цены с комментарием.

- `option_price_rub`
  Цена опции в рублях, если её удалось однозначно извлечь.
  Если поле пустое, это может означать, что у опции не было отдельной цены, цена не была указана на листе или значение было нечисловым.

- `notes`
  Дополнительные пояснения по статусу или цене опции.
  Используется для сохранения смысла, который не помещается в основные поля.

- `row_order`
  Порядок строки опции на листе Excel.
  Полезен для сохранения исходного порядка отображения.

- `column_order`
  Порядок колонки модели на листе Excel.
  Полезен для восстановления исходного порядка моделей на листе.

### Как использовать таблицу `comparison_model_options`

- Для ответа по комплектации модели нужно собирать все строки по `comp_full_name`, а не смотреть одну запись.
- Одна и та же опция может встречаться в нескольких листах и файлах. Поэтому по модели нужно агрегировать статусы и цены по нескольким строкам.
- Если цена допопции отсутствует в одной записи, она может присутствовать в другой записи этой же модели или на другом листе.
- Для поиска опций сначала надо фильтровать по `comp_full_name`, затем по `option_name`, и только потом анализировать статус и цену.

## Таблица `comparison_service_groups`

Таблица содержит данные по техническому обслуживанию: интервалы ТО, пробеги и стоимости сервисных работ для групп моделей.

### Поля

- `source_file`
  Исходный Excel-файл, из которого получена запись по ТО.

- `source_sheet`
  Исходный лист Excel с данными по ТО.

- `base_model`
  Модель GAZ, относительно которой выполняется сравнение на листе ТО.
  Это контекст сравнения, а не базовая версия автомобиля.

- `group_name`
  Название группы моделей или семейства, для которого указаны значения ТО.
  Это не обязательно конкретная модель; иногда это группа или объединённая категория моделей на листе.

- `service_label`
  Текстовое имя конкретного ТО или сервисной операции.
  Например: `ТО-1`, `ТО-2` и аналогичные обозначения из Excel.

- `service_index`
  Порядковый номер ТО, если он был извлечён из обозначения.

- `mileage_km`
  Пробег, на котором выполняется соответствующее ТО, в километрах.

- `cost_diesel_rub`
  Стоимость ТО для дизельной версии в рублях.

- `cost_diesel_promo_rub`
  Стоимость ТО для дизельной версии по промо-цене в рублях, если такая цена была указана.

- `cost_gasoline_rub`
  Стоимость ТО для бензиновой версии в рублях.

- `cost_gasoline_promo_rub`
  Стоимость ТО для бензиновой версии по промо-цене в рублях, если такая цена была указана.

- `notes`
  Дополнительные примечания по строке ТО, если на листе присутствовали особые условия или пояснения.

### Как использовать таблицу `comparison_service_groups`

- Эта таблица описывает не полные технические характеристики модели, а только данные по ТО.
- Ответы по стоимости обслуживания надо строить по `group_name`, `service_label` и `mileage_km`.
- Если одна и та же группа встречается несколько раз, нужно агрегировать записи и учитывать все непустые значения стоимости.

## Практические рекомендации для запросов к базе

1. Не ищи информацию по модели в одной строке. Всегда собирай несколько записей по одному `comp_full_name`.
2. Если у записи нет цены, не делай вывод, что цена неизвестна. Проверь другие записи той же модели в `comparisons_normalized`.
3. Все фильтры и сопоставления по текстовым полям выполняй через зеркальные поля `_nocase`.
4. Если вопрос про допоборудование, комплектацию или наличие опции, используй `comparison_model_options`.
5. Если вопрос про ТО или сервисные затраты, используй `comparison_service_groups`.
6. Если одна и та же модель встречается в нескольких контекстах сравнения, объединяй данные по модели, а `base_model` используй как контекст того, с какой моделью GAZ сравнивали эту запись.
7. Не пытайся искать или использовать поле `comp_model`: в рабочей схеме этой базы его нет.
