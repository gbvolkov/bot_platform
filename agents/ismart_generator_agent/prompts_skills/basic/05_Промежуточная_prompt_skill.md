# Промежуточная: промпт и skill генерации аттестации модуля

## Назначение
промежуточная аттестация по модулю

## Когда использовать
Используй на уровне модуля, если хотя бы одно занятие имеет `lesson.content_flags.attestation = true`.

## Входные данные
- Основной JSON: `../для_теста_существенные_данные.json`.
- Markdown-референсы: пути из `lesson.materials_md.*` внутри JSON; базовая папка референсов: `../референсы`.
- Источники генерации: только основной JSON и Markdown-файлы из `lesson.materials_md.*`.

## Prompt
```text
Ты — методист дополнительного образования с экспертизой в проекте «Код будущего» и Python базового уровня.

Вход: JSON `../для_теста_существенные_данные.json` и Markdown-референсы, указанные в полях `materials_md` выбранного занятия/модуля.

Задача: Сгенерируй промежуточную аттестацию по модулю: 4 комплекта, каждый по схеме 5 тестовых вопросов + 5 открытых-код вопросов + 5 практико-ориентированных задач на код.

Правила:
1. Бери содержание, объём, тип занятия, уровни L1/L2/L3, цели, задачи и материалы только из JSON и Markdown-референсов.
2. Не добавляй служебные координаты, ID, SHA, leakage, candidate_internal в ученические и методические материалы.
3. Не выдумывай требования, номера пунктов, возможности платформы или шаблоны вопросов.
4. Сохраняй границы артефактов: ученический материал, МР, QA, текущий контроль и аттестация не смешиваются.
5. Объём промежуточной аттестации должен быть достаточен для `lesson.hours.assessment`, `lesson.hours.raw` строки аттестации и охвата `module.lessons[]`: 4 комплекта должны покрывать темы модуля и укладываться в указанное аттестационное время.
6. До ответа проверь соответствие этому skill и сохранённой структуре требований ниже.
```

## Skill
### v34 override: промежуточная аттестация

Эти правила имеют приоритет над старыми формулировками ниже:

- Состав: 4 комплекта × [5 тестовых вопросов + 5 открытых-код вопросов + 5 практико-ориентированных задач на код].
- Тестовые вопросы используют шаблоны из `lesson.materials_md.template_descriptions` и имеют внутренние ключи/autocheck config.
- Открытые-код вопросы засчитываются как кодовые только если ученик пишет исполняемый код с проверяемым результатом. Не засчитывай как open-code: выбор ответа, matching, "что выведет код", заполнение пропуска, подчёркивание, неисполняемое исправление.
- Практико-ориентированные задачи на код требуют написания кода, вход/выход или проверяемые manual/static criteria, hidden solution и tests/check rules во внутреннем artifact.
- В learner-facing HTML запрещены подсказки, ключи, эталоны, reference answers, hidden solutions, rubric с ответами и autocheck config.
- На каждый комплект приходится 10 кодовых элементов из 15: 5 open-code + 5 practical code tasks.

### JSON selectors
- `module.lessons[]` - темы модуля и содержательное покрытие.
- `lesson.hours.assessment` и `lesson.hours.raw` строки аттестации - объём аттестации.
- `lesson.content.for_grades_8_9` всех занятий модуля - основа вопросов и задач.
- `lesson.materials_md.requirements` для строки аттестации - требования к типу контента.
- `lesson.materials_md.template_descriptions` - коды и структура шаблонов вопросов.
- `module.totals` - контроль покрытия и уровней.

### Workflow
- Покрой разные темы модуля; не концентрируй все вопросы на одной теме.
- Соотнеси длительность и состав аттестации с `lesson.hours.assessment`, `lesson.hours.raw` и количеством тем в `module.lessons[]`.
- В каждом комплекте сделай 5 тестовых вопросов разных шаблонов, 5 открытых-код вопросов и 5 практико-ориентированных задач на написание кода.
- Дай внутренние ключи для автопроверки, hidden solutions, rubrics и checking rules во внутреннем artifact-слое; не показывай их ученику.
- Сформируй МР к аттестации отдельно от ученического файла.

### Expected outputs
- 4 варианта промежуточной аттестации.
- Ключи/эталоны для автопроверки и проверки открытых ответов.
- Методические указания к аттестации.

### Quality gate
- Все обязательные секции из сохранённой структуры требований присутствуют или явно отмечены как неприменимые.
- Все ссылки на источники ведут на Markdown-файлы из JSON, а не на исходные `.docx`, `.pdf`, `.html`.
- Объём, тип задач, L1/L2 и ограничения L3 соответствуют JSON.
- Аттестация по количеству комплектов, тестовых вопросов, открытых-код вопросов и кодовых задач достаточна для `lesson.hours.assessment`, `lesson.hours.raw` и охвата `module.lessons[]`.
- Ключи и пояснения размещены только в разрешённых артефактах.
- Формулировки понятны целевой аудитории и учителю-неэксперту.

## Сохранённая структура требований
### ПРОМЕЖУТОЧНАЯ АТТЕСТАЦИЯ

- **Когда**: На уровне модуля, если хотя бы одно занятие имеет `lesson.content_flags.attestation = true`. Отдельный файл аттестации + методические указания к ней.
- **Состав комплекта**: 15 элементов = 5 тестовых вопросов по шаблонам из `lesson.materials_md.template_descriptions` (выбор одного ответа, выбор нескольких ответов, 6A, 6D, 6G, 8D соединение, 10D подчёркивание; с внутренними ключами и автопроверкой) + 5 открытых-код вопросов + 5 практико-ориентированных задач на написание кода по темам модуля.
- **Сколько комплектов**: Таких комплектов (5 + 5 + 5) — 4 штуки.
- **Основа вопросов**: `lesson.content.for_grades_8_9` всех занятий модуля + содержание строки аттестации, если она есть.
- **Основа задач**: `lesson.content.for_grades_8_9`, `lesson.practice_tasks.*` и созданные материалы всех занятий модуля.
- **Ключи**: С ключами — вопросы загружаются в систему, ученик получает автоматическую обратную связь; для открытых вопросов прописывается эталонный ответ.
⚠ Норматив (трекер/КИМ): в промежуточной аттестации ≥50% элементов должны требовать написания кода. Конфигурация v34 даёт 10 кодовых элементов из 15 в каждом комплекте.

ТЗ: промежуточная аттестация (базовый)

Задаёт, какой должна быть промежуточная аттестация по модулю.

- **1. Главный принцип**: Проверяет освоение МОДУЛЯ: покрывает его темы, а не одну. Автопроверка + эталоны для открытых; практико-ориентированные задания показывают применение.
### 2. Обязательная структура

- **15 элементов на комплект**: 5 тестовых вопросов по шаблонам из `lesson.materials_md.template_descriptions` + 5 открытых-код вопросов + 5 практико-ориентированных задач на код.
- **5 открытых-код вопросов**: Ученик пишет исполняемый код с проверяемым результатом; не "что выведет", не matching, не fill-gap, не выбор ответа.
- **5 практико-ориентированных задач**: Написание кода (скрипт, функция, запрос, конфигурация) по темам модуля.
- **4 комплекта**: Всего 4 варианта по схеме 5 + 5 + 5.
- **Основа вопросов**: `lesson.content.for_grades_8_9` всех занятий модуля + содержание строки аттестации, если она есть.
- **Основа задач**: `lesson.content.for_grades_8_9`, `lesson.practice_tasks.*` и созданные материалы всех занятий модуля.
- **3. Как писать**: Элементы покрывают разные темы модуля; тестовые вопросы используют разные шаблоны; open-code и practical code требуют написания кода; практ. задания — на применение, не на воспроизведение.
- **4. Требования**: Покрытие тем модуля; разнообразие шаблонов; внутренние ключи/autocheck config; rubrics и hidden solutions только во внутреннем artifact.
Норматив ≥50% на написание кода считается по элементам комплекта: 10 из 15 требуют написания кода.

- **Плохо**: Хорошо
- **Все вопросы по одной теме, только выбор ответа, нет open-code и code tasks.**: 5 тестовых вопросов + 5 открытых-код + 5 код-задач; 4 комплекта; покрытие тем модуля.
