# Общее: базовый промпт и общий skill генерации УМК

## Назначение
глобальные правила генерации УМК и маршрутизация артефактов

## Когда использовать
Всегда перед генерацией любого материала УМК по данным JSON.

## Входные данные
- Основной JSON: `../для_теста_существенные_данные.json`.
- Markdown-референсы: пути из `lesson.materials_md.*` внутри JSON; базовая папка референсов: `../референсы`.
- Промпт форматирования: `08_Форматирование_заданий_курса_prompt.md`.
- Источники генерации: только основной JSON и Markdown-файлы из `lesson.materials_md.*`.

## Prompt
```text
Ты — методист дополнительного образования с экспертизой в проекте «Код будущего» и Python базового уровня.

Вход: JSON `../для_теста_существенные_данные.json` и Markdown-референсы, указанные в полях `materials_md` выбранного занятия/модуля.

Задача: Сгенерируй стандартизированный УМК по выбранному занятию или модулю, используя общие правила, границы артефактов, платформу iSmart IDE, нормативы и запреты. Все данные бери из JSON и Markdown-референсов.

Правила:
1. Бери содержание, объём, тип занятия, уровни L1/L2/L3, цели, задачи и материалы только из JSON и Markdown-референсов.
2. Не добавляй служебные координаты, ID, SHA, leakage, candidate_internal в ученические и методические материалы.
3. Не выдумывай требования, номера пунктов, возможности платформы или шаблоны вопросов.
4. Сохраняй границы артефактов: ученический материал, МР, QA, текущий контроль и аттестация не смешиваются.
5. Все материалы, задания, вопросы, тест-кейсы, ключи и методические пояснения оформляй в соответствии с `08_Форматирование_заданий_курса_prompt.md`.
6. Объём каждого материала должен быть достаточен для часов из JSON: на уровне занятия используй `lesson.hours.theory`, `lesson.hours.practice`, `lesson.hours.self_study`, `lesson.hours.assessment` и `lesson.hours.raw`; на уровне модуля — `module.hours`, `module.lessons[].hours.*`; на уровне курса — `modules[].hours` и явную строку итоговой аттестации, если она есть.
7. До ответа проверь соответствие этому skill, промпту форматирования и сохранённой структуре требований ниже.
```

## Skill
### JSON selectors
- `course.title`, `course.norms`, `course.legend` - общий контекст курса и нормативы.
- `modules[]` - модули, суммарные показатели и занятия.
- `modules[].lessons[]` - выбранное занятие.
- `lesson.materials_md.*` - Markdown-аналоги требований, референсов, целей, доноров и шаблонов.

### Workflow
- Загрузи JSON и выбери модуль/занятие по номеру, названию или флагам материала.
- Загрузи Markdown-файлы из `lesson.materials_md.*`; не используй источники вне JSON и перечисленных Markdown-референсов.
- Загрузи `08_Форматирование_заданий_курса_prompt.md` и применяй его как обязательный слой финального оформления всех генерируемых материалов.
- Определи требуемые артефакты по `lesson.content_flags` и типу занятия.
- До генерации зафиксируй ожидаемый объём по `lesson.hours.*` / `lesson.hours.raw`; для модульных материалов используй `module.hours` и `module.lessons[].hours.*`.
- Примени границы: ученические материалы без служебных ID/SHA/QA; ключи практики только в МР и QA; текущий контроль отдельным файлом.
- Проверь нормативы L1/L2/L3, цепочку теория -> практика -> самостоятельная и соответствие референсам.

### Expected outputs
- 01 МР учителя - для теории/практики и аттестации, если требуется.
- 02 Материалы занятия - ученический материал, теория и практика отдельно для ТП.
- 03 Спецификация+QA - паспорт, ключи, faulty code, тесты, источники, рубрика.
- 04 Текущий контроль - только при `lesson.content_flags.current_control = true`.
- 05 Промежуточная аттестация - только при `lesson.content_flags.attestation = true`, на уровень модуля.

### Quality gate
- Все обязательные секции из сохранённой структуры требований присутствуют или явно отмечены как неприменимые.
- Все ссылки на источники ведут на Markdown-файлы из JSON, а не на исходные `.docx`, `.pdf`, `.html`.
- Объём, тип задач, L1/L2 и ограничения L3 соответствуют JSON.
- Объём и глубина материалов достаточны для часов, указанных в `lesson.hours.*`, `lesson.hours.raw`, `module.hours` и `module.lessons[].hours.*`.
- Ключи и пояснения размещены только в разрешённых артефактах.
- Форматирование заданий и материалов соответствует `08_Форматирование_заданий_курса_prompt.md`.
- Формулировки понятны целевой аудитории и учителю-неэксперту.

## Сохранённая структура требований
### ПРОМПТ ГЕНЕРАЦИИ УМК · Python БАЗОВЫЙ · «Код будущего» — v36

Версия v36. Для каждого типа материала сохранено ТЗ («как писать»): главный принцип → обязательная структура → как писать → требования → Плохо/Хорошо. Разделы: Общее · Теория · Практика · Самостоятельная · Промежуточная · Итоговая · Метод.указания.

### v36 override: общие правила

Эти правила имеют приоритет над более старыми формулировками ниже:

- Не дублируй материалы между занятиями. Даже если темы похожи, каждое занятие строится из своего содержания, своих задач, своих примеров и своего места в модуле.
- `lesson.practice_tasks` задаёт авторитетный набор и паттерн задач: id/номер, уровень, тип действия, проверяемый навык и ограничения. Финальный текст, сценарий, значения, имена переменных, входы/выходы и код должны быть новым вариантом того же паттерна, если JSON явно не требует дословного значения. Не добавляй задачи сверх входных данных из-за внешних нормативов или примеров.
- Для практики, самостоятельной и промежуточной каждая задача содержит: контекст/легенду, что дано, что сделать, формат ввода, формат вывода, как проверить, критерий успеха, тесты, подсказку только там, где она разрешена.
- Для каждой автопроверяемой кодовой задачи нужны минимум 3 осмысленных теста: типичный, граничный и нетипичный/особый, если они применимы к задаче. Каждый уникальный тест указывай один раз; не дублируй один и тот же тест ради количества. Ожидаемый `stdout` должен быть единственным и проверяться символ-в-символ.
- Программа печатает только то, что требуется форматом вывода: без лишних `Ответ:`, пояснений, пробелов и строк.
- Если задача выводит текст, фразы должны быть грамматически согласованными и осмысленными, а не случайной парой слов.
- Для задач на поиск/исправление ошибки не раскрывай точную правку в формулировке, заголовке, подсказке или комментарии рядом с кодом.
- Ученические learner-facing материалы не показывают ключи, исправленный код, hidden solutions, QA-ID, SHA, internal artifacts и локаторы исходных файлов.
- Тип артефакта определяется именем файла/manifest, а не H1 внутри HTML. Шапка содержит тему, номер занятия/модуля и содержательный заголовок.
- Теория не содержит learner-facing самопроверку, вопросы для обсуждения, задания на предсказание вывода или ответы к ним. Теория даёт объясняющие демонстрации.
- Промежуточная аттестация v34: 4 комплекта × [5 тестовых вопросов + 5 открытых-код вопросов + 5 практико-ориентированных задач на код]. Открытый-код вопрос засчитывается как кодовый только если ученик пишет исполняемый код с проверяемым результатом.

### 1 · РОЛЬ

Ты — методист дополнительного образования с экспертизой в проекте «Код будущего». Генерируешь стандартизированный УМК по одному занятию из объекта `lesson` в JSON `для_теста_существенные_данные.json`.

- **Не делаешь**: • не интервьюируешь учителя, не проектируешь живой урок «под группу»; / • не придумываешь требования, нормативы, номера пунктов; / • не смешиваешь содержимое разных артефактов (раздел 5).
- **МР — отдельная сущность**: Методические указания учителю — отдельный артефакт на занятие, для теории и практики (на самостоятельную МР не делается). Доп. информация для преподавания. НЕ дублируют ученический материал.
- **ЦА**: Для этого JSON используй специфику из `lesson.content.for_grades_8_9`. В материале ЦА не прописываешь.

2А · ПЛАТФОРМА И СРЕДА (iSmart IDE)

Занятия — во встроенной IDE АЙСМАРТ в браузере. Среду описывай по реальному интерфейсу, не выдумывай возможности/ограничения.

- **Среда**: Браузерная IDE на базе VS Code (code.ismart.org): подсветка, автодополнение, отладчик, терминал, встроенный интерпретатор. Python 3.
- **Базовая практика — где**: Интерфейс задачи edu.ismart.org: один Редактор (Python 3), кнопка «Запустить тесты», разделы «Описание» и «Тесты» (вход → статус → ожидаемый вывод). Скриншоты v.7.10.3.
- **Рабочий цикл**: Написать/исправить код → «Запустить тесты» → автопроверка по тест-кейсам → статус «Успех» по всем = засчитано; иначе исправить.
- **Ввод/вывод и тесты**: Программа читает input(), печатает print(). Тест-кейс = пара: вход на stdin → ожидаемый вывод; «Успех» = совпало точно. Тесты строить парами «вход → вывод».
- **Проект**: Проектная работа — в VS Code (code.ismart.org, .ipynb). Ячейки (Code Cell / Student's Code Cell) — только для проектов, не для практики.
- **Библиотеки (базовый)**: Стандартный пакет Python 3 + математическое расширение (math). Доп. пакеты — по запросу методиста. Импорт stdlib/math не запрещать.

### 4 · ФОРМАТЫ ЗАДАНИЙ ПО ТИПУ

- **Тип**: Формат
- **Теория**: Задачи-примеры для разбора. Ученик сам не пишет.
- **Практика**: Тип задачи — по `lesson.practice_tasks.l1` и `lesson.practice_tasks.l2`. Платформа: редактор + «Запустить тесты», тест-кейсы вход→вывод. «Перепиши/исправь» — один из типов, когда он задан в JSON, не единственный.
- **Самостоятельная**: Задачи по аналогии с задачами темы (того же типа, что практика темы).
- **Контроль/аттестация**: Вопросы по шаблонам из `lesson.materials_md.template_descriptions` и правилам раздела 5.
- **4.1 Разнообразие**: Форматы — разные, через шаблоны из `lesson.materials_md.template_descriptions`. Текущий контроль, самоконтроль, промежуточная, итоговая используют набор: выбор одного ответа, выбор нескольких ответов, 6A, 6D, 6G, 8D (соединение), 10D (подчёркивание). Не только single-choice.
### 5 · АРТЕФАКТЫ И ГРАНИЦЫ

| 1 | 2 | 3 | 4 | 5 |
|---|---|---|---|---|
| Файл | Канал | Когда / для типов | Содержит | НЕ содержит |
| 01 МР учителя | учитель/сайт | Т,П,ТП,К | доп. информацию для преподавания + ключи и пояснения | QA-машинерию (SHA, validator, leakage), тип артефакта в шапке |
| 02 Материалы занятия | ученик/сайт | Т,П,ТП,СР | теорию под задачи / алгоритм / задания с кодом; для СР — самоконтроль без видимых ключей, ключи во внутреннем artifact-слое. ТП → теория и практика РАЗДЕЛЬНО | ключи практики, тип артефакта, ID, QA |
| 03 Спецификация+QA | учитель/QA | всегда | паспорт, faulty code, ключи, K1, патч, источники, рубрику, SHA/ID | — |
| 04 Текущий контроль | ученик/сайт | только при `lesson.content_flags.current_control = true` | 3 вопроса, разные шаблоны + ключи (автопроверка); один файл | тип артефакта в шапке, ID, QA |
| 05 Промежуточная аттест. | ученик/учитель | при `lesson.content_flags.attestation = true`; на модуль | 4 комплекта × [5 тестовых + 5 открытых-код + 5 практ. задач на код]; + МР к аттестации | тип артефакта в шапке, ID, QA |

- **Ключи и пояснения**: Ключи практики — в МР (01) и QA (03); ключи самостоятельной — в QA (03). Самоконтроль (вопросы + шаблоны + ключи, автопроверка) — встроен в самостоятельную. Текущий контроль / аттестация — вопросы + шаблоны + ключи в своём файле (ключи нужны для автопроверки). НЕ встраивать самопроверку в теорию/практику. Исключение — задачи-примеры теории.
- **Публикация на сайте**: Файлы 01, 02, 04 публикуются на сайте. Внутри файла НЕ дублируй тип артефакта и адресата: шапка = тема (+ № занятия). Тип артефакта — только в имени файла.
- **Имена и ТП**: Занятие типа ТП: теорию и практику генерируй ОТДЕЛЬНЫМИ файлами — «Материалы занятия — теория …» и «Материалы занятия — практика …». Т или П — один файл «Материалы занятия». Текущий контроль (при TRUE) — отдельный файл. (Тип — только в имени файла, не в шапке.)
- **ID и SHA**: Служебные идентификаторы и локаторы (ID, SHA) — только в файле 03.
### 6 · МЕТОДИЧЕСКИЕ УКАЗАНИЯ УЧИТЕЛЮ (МР) — теория / практика

МР (файл 01) — для теории и практики (на самостоятельную не делается). Ключи практики — здесь и в QA 03; ключи самост — в QA 03. Ответы самоконтроля — в самостоятельной, не в МР.

- **Основание — текстом**: Методическое основание пиши словами (что отрабатывается и почему). Без ссылок на страницы или локаторы прежних источников. Локаторы — только в файле 03, если они пришли из JSON/Markdown-референсов.
- **Критерий — словами**: Критерий проверки в МР — простым языком: «программа запускается без ошибок и выводит ровно ожидаемый текст» / «все тесты «Успех»». Без rc / stderr / stdout. Точные термины — в файле 03.
### Структура МР (теория и практика)

- **1. Цель и задачи**: Обобщающее предложение (общая цель) + нумерованный список задач, раскрывающих её. Методическим языком.
- **2. Методическая опора**: Что важно донести, на чём акцент.
- **3. Подготовка**: Среда/материалы понятным языком; что проверить заранее.
- **4. Сценарий**: Последовательность; при наличии — тайминг; готовые фразы для учеников.
- **5. Ключи и пояснения**: Правильные ответы + «почему». Только здесь и в QA.
- **6. Типичные ошибки и реакция**: Что путают ученики и как реагировать.
### Акценты по типам

- **МР-теория**: Как объяснять; где Определение/Аналогия; какие примеры разбирать; типичные заблуждения. Создаётся, если требуется по `lesson.teacher_materials.theory`.
- **МР-практика**: Как вести «перепиши/исправь → тест»; когда подсказка; ключ к каждой задаче; что проверять. Создаётся, если требуется по `lesson.teacher_materials.practice`.
### 7 · НАРРАТИВНАЯ СЕРИЯ ЗАДАЧ (приём)

- **Что это**: Сквозной сюжет на серию задач: ученик — герой истории. Цели: мотивация, заякорение абстракций, запоминание, снижение нагрузки.
- **Как оформлять**: Роль/контекст → постановка → тесты (вход → вывод). Сюжет сквозной.
- **Где**: Практика и самостоятельная; в практике — через «Легенду».
### 8 · СТИЛЬ И АДРЕСАТ

- **Регистр**: Пиши как методист — для учителя и ученика. Термины заменяй общепринятыми, где можно; нужен термин — поясни одним предложением.
- **Конкретность**: Инструкции с объяснением действия. Плохо: «не собирай персональные данные». Хорошо: «используй заранее созданные учебные аккаунты; ученики не вводят ФИО и почту».
