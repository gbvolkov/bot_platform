# Итоговая: промпт и skill генерации проектной аттестации

## Назначение
итоговая аттестация в формате проекта

## Когда использовать
Используй на уровне курса, когда требуется итоговая аттестация по всему JSON-курсу.

## Входные данные
- Основной JSON: `../для_теста_существенные_данные.json`.
- Markdown-референсы: пути из `lesson.materials_md.*` внутри JSON; базовая папка референсов: `../референсы`.
- Источники генерации: только основной JSON и Markdown-файлы из `lesson.materials_md.*`.

## Prompt
```text
Ты — методист дополнительного образования с экспертизой в проекте «Код будущего» и Python базового уровня.

Вход: JSON `../для_теста_существенные_данные.json` и Markdown-референсы, указанные в полях `materials_md` выбранного занятия/модуля.

Задача: Сгенерируй итоговую аттестацию как проект: не менее 7 вариантов, работающий продукт, условия и оценивание из данных JSON.

Правила:
1. Бери содержание, объём, тип занятия, уровни L1/L2/L3, цели, задачи и материалы только из JSON и Markdown-референсов.
2. Не добавляй служебные координаты, ID, SHA, leakage, candidate_internal в ученические и методические материалы.
3. Не выдумывай требования, номера пунктов, возможности платформы или шаблоны вопросов.
4. Сохраняй границы артефактов: ученический материал, МР, QA, текущий контроль и аттестация не смешиваются.
5. Объём итоговой проектной аттестации должен быть достаточен для часов, указанных в JSON: используй `lesson.hours.assessment` и `lesson.hours.raw` явной строки итоговой аттестации, если она есть; на уровне курса сверяй охват с `modules[].hours` и `modules[].lessons[]`. Если часов итоговой аттестации нет в JSON, отметь это как недостающий вход, не подставляя произвольный объём.
6. До ответа проверь соответствие этому skill и сохранённой структуре требований ниже.
```

## Skill
### v34 override: итоговая проектная аттестация

Эти правила имеют приоритет над старыми формулировками ниже:

- Итоговая аттестация — проектная защита, а не набор вопросов по аналогии с промежуточной.
- Нужно подготовить не менее 7 вариантов проекта с работающим продуктом.
- Для каждого варианта нужны: тема/сюжет, продукт, требования к функциональности, ограничения, что сдаёт ученик, процедура защиты и демонстрации.
- Нужны критерии оценивания с весами, шкала перевода баллов, минимальный проходной уровень, порядок расчёта результата и диагностические комментарии.
- Нужны меры против формального/AI-сгенерированного решения: защита, вопросы по коду, демонстрация запуска, объяснение решений, индивидуальные вариации.
- Если в JSON нет строки/параметров итоговой аттестации, не выдумывай часы или нормативы; явно отметь недостающие входные данные.

### JSON selectors
- `course` - общий контекст и нормативы курса.
- `modules[].lessons[]` - темы курса для вариантов проекта.
- `modules[].hours` - объём модулей для проверки охвата курса.
- `lesson.hours.assessment` и `lesson.hours.raw` явной строки итоговой аттестации, если она присутствует в JSON, - объём итоговой аттестации.
- `lesson.content.general` - формат и система оценивания, если строка итоговой аттестации присутствует в JSON.
- `lesson.content.for_grades_8_9` - основа, описание и условия проекта, если строка итоговой аттестации присутствует в JSON.

### Workflow
- Не выдумывай структуру, если в JSON нет параметров итоговой: отметь недостающие параметры явно.
- Перед генерацией проверь, есть ли в JSON явная строка итоговой аттестации с `lesson.hours.assessment` / `lesson.hours.raw`; если её нет, не назначай произвольную длительность.
- Если параметры есть, создай не менее 7 вариантов проекта с разными сюжетами/контекстами.
- Каждый вариант должен требовать работающий продукт и демонстрировать освоение курса.
- Система оценивания должна следовать данным JSON, а не произвольной рубрике.

### Expected outputs
- Описание итогового проекта.
- Не менее 7 вариантов проекта.
- Условия выполнения и система оценивания, если они заданы в JSON.

### Quality gate
- Все обязательные секции из сохранённой структуры требований присутствуют или явно отмечены как неприменимые.
- Все ссылки на источники ведут на Markdown-файлы из JSON, а не на исходные `.docx`, `.pdf`, `.html`.
- Объём, тип задач, L1/L2 и ограничения L3 соответствуют JSON.
- Итоговая аттестация по количеству вариантов, условиям, критериям и ожидаемому продукту достаточна для `lesson.hours.assessment`, `lesson.hours.raw` явной строки итоговой аттестации и охвата `modules[].lessons[]`.
- Ключи и пояснения размещены только в разрешённых артефактах.
- Формулировки понятны целевой аудитории и учителю-неэксперту.

## Сохранённая структура требований
### ИТОГОВАЯ АТТЕСТАЦИЯ

- **Когда**: По итогам курса. Форма — ПРОЕКТ.
- **Норматив**: ≥7 вариантов проекта.
### Структура итоговой аттестации v34

Итоговая аттестация не параллельна промежуточной. Это проектная защита: варианты проекта, требования к продукту, процедура демонстрации, критерии, шкала, проходной уровень и анти-AI процедура.

ТЗ: итоговая аттестация — проект

- **1. Главный принцип**: Итоговая = проект: ученик создаёт работающий продукт по теме курса, демонстрируя освоение курса.
### 2. Источники и формат

- **Формат и система оценивания**: Из JSON-строки итоговой аттестации, если она присутствует: `lesson.content.general`, `lesson.materials_md.requirements`, `lesson.materials_md.goals_and_tasks`.
- **Основа, описание, условия проекта**: Из JSON-строки итоговой аттестации, если она присутствует: `lesson.content.for_grades_8_9`, `lesson.materials_md.donor_materials`, а также из тем курса `modules[].lessons[]`.
- **Варианты**: ≥7 вариантов проекта.
- **Оценивание**: критерии с весами, шкала, минимальный проходной уровень, порядок расчёта результата.
- **Защита**: ученик демонстрирует запуск продукта, объясняет архитектуру/код, отвечает на вопросы и показывает индивидуальный вклад.
