# Теория: промпт и skill генерации объясняющего материала

## Назначение
ученический материал теории и МР-теория

## Когда использовать
Используй для занятий, где `lesson.content_flags.theory = true`; для ТП генерируй теорию отдельным файлом.

## Входные данные
- Основной JSON: `../для_теста_существенные_данные.json`.
- Markdown-референсы: пути из `lesson.materials_md.*` внутри JSON; базовая папка референсов: `../референсы`.
- Источники генерации: только основной JSON и Markdown-файлы из `lesson.materials_md.*`.

## Prompt
```text
Ты — методист дополнительного образования с экспертизой в проекте «Код будущего» и Python базового уровня.

Вход: JSON `../для_теста_существенные_данные.json` и Markdown-референсы, указанные в полях `materials_md` выбранного занятия/модуля.

Задача: Сгенерируй объясняющий теоретический материал для ученика и, если требуется, МР-теорию для учителя. Теория должна объяснять почему и как, а не перечислять факты. Материал обязан раскрывать основные используемые понятия и содержание темы: что изучается, зачем это нужно, как это работает, как выглядит в коде и где ученик обычно ошибается.

Правила:
1. Бери содержание, объём, тип занятия, уровни L1/L2/L3, цели, задачи и материалы только из JSON и Markdown-референсов.
2. Не добавляй служебные координаты, ID, SHA, leakage, candidate_internal в ученические и методические материалы.
3. Не выдумывай требования, номера пунктов, возможности платформы или шаблоны вопросов.
4. Сохраняй границы артефактов: ученический материал, МР, QA, текущий контроль и аттестация не смешиваются.
5. Ученический материал "Теория" всегда строится по явной структуре: шапка, введение/связка с предыдущим, цель занятия, задачи занятия, ключевые понятия, объясняющее содержание темы, разобранные демонстрации, типичные ошибки, важные моменты, итог и мостик к практике. Нельзя заменять эти разделы только оглавлением или общим конспектом.
6. Объём теоретического материала должен быть достаточен для `lesson.hours.theory` и `lesson.hours.raw`: при 1 академическом часе дай полноценный объясняющий конспект, демонстрации, типичные ошибки и выводы; если `lesson.hours.theory = 0`, теорию не генерируй как отдельный ученический материал.
7. До ответа проверь соответствие этому skill и сохранённой структуре требований ниже.
```

## Skill
### JSON selectors
- `lesson.content.general` и `lesson.content.for_grades_8_9` - каркас темы.
- `lesson.hours.theory` и `lesson.hours.raw` - объём.
- `lesson.materials_md.requirements` - требования к теории.
- `lesson.materials_md.reference_examples` - эталон структуры и оформления.
- `lesson.materials_md.goals_and_tasks` - цели и задачи, не сочинять заново.
- `lesson.materials_md.donor_materials` - фактура и примеры.
- `lesson.teacher_materials.theory` - сигнал к генерации МР-теории.

### Workflow
- Выдели проблему/зачем нужна тема и свяжи с предыдущими темами, если это не первое занятие.
- Спланируй глубину объяснения по `lesson.hours.theory` и `lesson.hours.raw`; объём не должен выглядеть как краткая справка для часового занятия.
- Построй ученическую теорию как документ с обязательными разделами: "Цель занятия", "Задачи занятия", "Ключевые понятия", смысловые разделы конспекта, "Разобранные демонстрации", "Типичные ошибки", "Важные моменты", "Итоговые выводы" и "Переход к практике".
- В смысловых разделах конспекта раскрывай основные используемые понятия и содержание темы: сначала зачем это нужно, затем простое объяснение, затем точное определение, затем рабочий пример с выводом, затем пояснение типичной ошибки.
- Примеры должны доказывать мысль; показывай результат выполнения кода явно.
- В ученическом материале не размещай методические задачи и служебные локаторы.
- Для МР-теории используй общий skill методических указаний: акценты объяснения, подготовка, сценарий, ключи к задачам-примерам.

### Expected outputs
- Материал занятия - теория: шапка, цель занятия для ученика, задачи занятия для ученика, ключевые понятия, объясняющий конспект с раскрытием содержания темы, разобранные демонстрации, типичные ошибки, важные моменты, итоговые выводы и мостик к практике.
- МР-теория: цель и задачи, методическая опора, подготовка, сценарий, ключи и пояснения, типичные ошибки.

### Quality gate
- Все обязательные секции из сохранённой структуры требований присутствуют или явно отмечены как неприменимые.
- В ученической теории явно есть разделы "Цель занятия", "Задачи занятия", "Ключевые понятия", "Типичные ошибки"; требования AB нельзя считать выполненными, если эти элементы только подразумеваются в тексте.
- Основные используемые понятия и содержание темы раскрыты в тексте, а не только перечислены: для каждого ключевого понятия есть понятное объяснение смысла, назначение, пример использования и связь с темой занятия.
- Все ссылки на источники ведут на Markdown-файлы из JSON, а не на исходные `.docx`, `.pdf`, `.html`.
- Объём, тип задач, L1/L2 и ограничения L3 соответствуют JSON.
- Теория по глубине и количеству смысловых блоков достаточна для `lesson.hours.theory` и `lesson.hours.raw`.
- Ключи и пояснения размещены только в разрешённых артефактах.
- Формулировки понятны целевой аудитории и учителю-неэксперту.

## Сохранённая структура требований
### ТЕОРИЯ — алгоритм и структура

### v34 override: теория

Эти правила имеют приоритет над старыми формулировками ниже:

- Теория не содержит learner-facing заданий, вопросов для обсуждения, предсказания вывода, блоков "Проверка себя" и ответов к ним.
- Теория является артефактом-изложением. Все примеры — разобранные демонстрации с объяснением, а не упражнения для ученика.
- Перед генерацией теории учитывай `lesson.practice_tasks` как описание будущих практических паттернов: теория должна заранее объяснить знания и типичные ошибки, которые потребуются в практике, но не раскрывать решения практических задач.
- Каждый содержательный пункт занятия раскрывается отдельным смысловым блоком. Если пункт не раскрыт, теория неполная.
- Для каждой демонстрации замыкай петлю: код → фактический результат выполнения → правило. Пример должен доказывать объясняемую мысль, а не просто иллюстрировать синтаксис.
- Для каждой типичной ошибки показывай неправильный код или неверное рассуждение, фактическое поведение Python, исправленный вариант и объяснение, почему исправление работает. Контраст «что получилось vs что ожидали» обязателен.
- Новое понятие вводи через контраст: было/стало, неверно/верно, ожидали/получили.
- Блок «Важные моменты» должен собирать все ключевые правила занятия, а не быть короткой выжимкой.
- Подстраивай язык под ЦА из JSON: для 8–9 проще и пошаговее, для 10–11 формальнее, для СПО прикладнее. Смысловую полноту не сокращай.
- В конце явно покажи, какие знания из теории будут применяться в практических заданиях, без готовых решений этих заданий.

### Алгоритм генерации

- **Шаг 1**: Требования по теории бери из `lesson.materials_md.requirements`: цели, задачи, ключевые понятия, типичные ошибки. Эти четыре группы должны попасть в ученический материал как явные разделы, а не раствориться внутри общего текста.
- **Шаг 2**: Каркас темы бери из `lesson.content.general` и `lesson.content.for_grades_8_9`; объём — из `lesson.hours.theory` и `lesson.hours.raw`.
- **Шаг 3**: Фактуру бери из Markdown-файлов, перечисленных в `lesson.materials_md.donor_materials`.
- **Шаг 4**: Цели и задачи бери из `lesson.materials_md.goals_and_tasks` и содержательных полей выбранного `lesson`; формулируй операционально, без ссылок на внешние источники.
- **Шаг 5**: Конспект строй по Markdown-референсам из `lesson.materials_md.reference_examples` + задачи-примеры для разбора.
- **Шаг 6**: Если `lesson.teacher_materials.theory` требует МР, заполни МР-теорию по `07_Методические_указания_prompt_skill.md`.
- **Выход**: Материал теории (ученик) + МР-теория (учитель), если она требуется по `lesson.teacher_materials.theory`.
### Структура документа (референс теория_2.1.1)

- **1. Шапка**: Занятие № + название; Модуль/Тема. Без ID.
- **2. Цель занятия**: Обязательный раздел в ученическом материале. Доступным языком: «После этого занятия я смогу: …» — практические умения по теме. Без жаргона.
- **3. Задачи занятия**: Обязательный раздел в ученическом материале. 3-5 операциональных задач ученика: что он научится делать, объяснять, запускать, проверять или исправлять. Не путать с методическим сценарием учителя.
- **4. Ключевые понятия**: Обязательный раздел в ученическом материале. Дай список основных понятий темы; для каждого понятия укажи краткое объяснение простым языком. Эти же понятия затем должны быть раскрыты в конспекте через смысл, пример и связь с темой.
- **5. Конспект лекции / содержание темы**: Разделы → подразделы. Каждый подраздел — связный объясняющий текст (зачем, как работает, разбор примера), а не строка-определение. Внутри: Определение, Ключевая идея, Аналогия, Примеры с разбором; таблицы где уместно. Конспект обязан раскрывать основные используемые понятия и содержание темы, а не только давать инструкции "что нажать" или "что написать".
- **6. Задачи-примеры / разобранные демонстрации**: Для разбора, с ответом (ключи допустимы — место преподавания). Каждый пример содержит код, фактический результат и правило, которое из него следует.
- **7. Типичные ошибки**: Обязательный раздел в ученическом материале. Дай 2-4 ошибки по теме: неверный код или неверное рассуждение, почему ошибка возникает, как её распознать и как исправить.
- **8. Важные моменты и мостик к практике**: Полный список правил занятия: что применить дальше, какие ошибки отслеживать, какие знания понадобятся в практике. Без решений практических задач.
- **9. Итоговые выводы**: Обобщение ключевых тезисов.
- **Глубина и стиль**: Развёрнутый объясняющий конспект уровня референса теория_2.1.1: связные абзацы, мотивация «зачем», пошаговый разбор примеров, типичные ошибки. Понятным языком, не телеграфно. Для ученика объясняй ДОСТУПНО и подробно: больше связного текста, простые примеры, без жаргона. Глубину масштабируй под тему и уровень.
ТЗ: объясняющий текст теории (базовый)

Задаёт, каким должен быть текст теории, чтобы он ОБЪЯСНЯЛ, а не перечислял факты. Цель базового уровня — чтобы тему понял тот, кто видит её впервые.

- **1. Главный принцип**: Теория объясняет «почему и как», а не констатирует «что запомнить». Текст ведёт от непонимания к пониманию. Провал: прочитал и не понял; формулировка ясна только тому, кто уже знает тему. Дело не в объёме — решает структура и наличие «зачем».
### 2. Обязательная структура темы

- **Связка с предыдущим**: Если тема опирается на пройденное — явная отсылка («на прошлом уроке… сегодня…»). Первая тема — пропускается.
- **Цель занятия**: Явно сформулируй, чему ученик научится после занятия. Не прячь цель в вводном абзаце.
- **Задачи занятия**: Явно перечисли действия ученика: понять, объяснить, написать, запустить, проверить, исправить. Не заменяй этот блок задачами-примерами.
- **Ключевые понятия**: Явно перечисли основные понятия темы и кратко объясни каждое. Дальше в тексте обязательно раскрой эти понятия через содержание, код и примеры.
- **Зачем это нужно / проблема**: Начни с задачи, которую решает понятие: ситуация → старый способ не работает → новый. Без «зачем» не запоминается.
- **Объяснение простым языком**: Раскрой суть до термина или сразу после. Термин — через смысл, не наоборот.
- **Определение**: Краткое строгое определение ПОСЛЕ того, как смысл понятен.
- **Разобранный пример**: Показывает разницу/эффект (см. п.6). Минимум 1–2 рабочих примера с комментариями.
- **Аналогия**: Перевод абстракции на знакомое из быта — для «ага»-момента.
- **Частые ошибки**: 1–3 типичные ошибки и почему возникают.
- **Мостик к практике**: Явная связь теории с будущими практическими паттернами без раскрытия решений. Самопроверка в теории не создаётся.
- **Заключение**: 2–4 тезиса, что унести с темы — опорные точки, не пересказ.
- **4. Логические цепочки**: Текст — маршрут, не список. Раскрытие понятия: определение → что значит → пример → аналогия. Маршрут по теме: видна общая линия. Связки-переходы между абзацами («раз умеем X, теперь Y», «но тут проблема…», «чтобы решить…»). Если абзацы переставимы без потери смысла — цепочки нет.
- **5. Как писать**: Операционально (создаёт/реализует/отлаживает). Под возраст ЦА (8–9 ближе к игре/быту, СПО — профконтекст). Учебный регистр, дружелюбно, без сюсюканья и рекламы. Термин объясняется при первом появлении. Связки между абзацами. Код рабочий, с комментариями, результат показан явно.
- **6. Требования к примерам**: Пример ДОКАЗЫВАЕТ мысль, а не противоречит ей. Перед вставкой спроси: показывает разницу/эффект — или прячет её?
- **Плохо**: Хорошо
- **print(21) и print("21") — оба выводят «21». Ученик решает, что разницы нет. Пример работает ПРОТИВ объяснения.**: print(2 + 2) → 4, а print("2" + "2") → 22. Одна операция ведёт себя по-разному — разница типов осязаема, пример доказывает мысль.

## Controlled advanced additions from Python advanced v16

Source: `Python advanced prompt workbook v16`, sheet `Theory`.

- Theory is an explanatory artifact: it must explain why and how the concept works, not only list what to remember.
- Do not add learner tasks, output-prediction exercises, discussion questions, or self-check blocks to theory; these belong to other material kinds.
- Before generation, use the lesson practice patterns as a boundary: theory must prepare exactly the knowledge, typical errors, and reasoning needed for those tasks without exposing their solutions.
- Each curriculum/topic point from the input data must be represented by a meaningful theory block; missing a source topic point means the theory is incomplete.
- Examples must be fully explained demonstrations with different neutral values from practice/reference tasks when overlap is possible.
