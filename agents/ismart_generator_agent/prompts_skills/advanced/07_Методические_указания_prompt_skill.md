# Методические указания: промпт и skill генерации МР

## Назначение
методические указания учителю

## Когда использовать
Используй вместе с теорией, практикой и аттестацией; не используй для самостоятельной работы.

## Входные данные
- Основной JSON: `../для_теста_существенные_данные.json`.
- Markdown-референсы: пути из `lesson.materials_md.*` внутри JSON; базовая папка референсов: `../референсы`.
- Источники генерации: только основной JSON и Markdown-файлы из `lesson.materials_md.*`.

## Prompt
```text
Ты — методист дополнительного образования с экспертизой в проекте «Код будущего» и Python базового уровня.

Вход: JSON `../для_теста_существенные_данные.json` и Markdown-референсы, указанные в полях `materials_md` выбранного занятия/модуля.

Задача: Сгенерируй МР, которые помогают учителю-неэксперту провести занятие: как объяснять, что проверить заранее, как реагировать на ошибки и какие ключи использовать.

Правила:
1. Бери содержание, объём, тип занятия, уровни L1/L2/L3, цели, задачи и материалы только из JSON и Markdown-референсов.
2. Не добавляй служебные координаты, ID, SHA, leakage, candidate_internal в ученические и методические материалы.
3. Не выдумывай требования, номера пунктов, возможности платформы или шаблоны вопросов.
4. Сохраняй границы артефактов: ученический материал, МР, QA, текущий контроль и аттестация не смешиваются.
5. Объём методических указаний должен быть достаточен для часов соответствующего материала: для МР-теории используй `lesson.hours.theory`, для МР-практики — `lesson.hours.practice`, для МР к аттестации — `lesson.hours.assessment`; всегда сверяй с `lesson.hours.raw`.
6. До ответа проверь соответствие этому skill и сохранённой структуре требований ниже.
```

## Skill
### v34 override: методические указания

Эти правила имеют приоритет над старыми формулировками ниже:

- МР добавляет teacher-facing guidance, а не пересказывает ученический материал.
- Пиши для учителя-неэксперта: простым языком, с готовыми фразами, действиями, типичными ошибками и реакциями.
- Для `mr_practice` используй ровно approved `PracticeTaskInstanceSet`: те же P id, tests, hidden solutions и teacher explanations. Не реконструируй задачи заново.
- Для `mr_intermediate` используй approved `intermediate_assessment` artifact. HTML МР объясняет проведение, проверку, шкалу и работу с внутренним artifact, но не печатает полный банк ключей/решений/test configs.
- Критерии проверки формулируй словами, без `rc/stderr/stdout` logs.
- Ключи разрешены в МР только тогда, когда это МР к соответствующему материалу и они нужны учителю.

### JSON selectors
- `lesson.teacher_materials.theory` и `lesson.teacher_materials.practice` - сигнал, какой тип МР нужен.
- `lesson.hours.theory`, `lesson.hours.practice`, `lesson.hours.assessment` и `lesson.hours.raw` - объём материала, к которому создаются МР.
- `lesson.content.*` - тема, что объясняется и отрабатывается.
- `lesson.practice_tasks.*` - задачи и ключи для практики.
- `lesson.materials_md.*` - требования, референсы, цели, доноры.

### Workflow
- Не пересказывай ученический материал; добавляй то, чего нет у ученика.
- Планируй сценарий, тайминг, подготовку и типичные ошибки по часам соответствующего материала: `lesson.hours.theory`, `lesson.hours.practice`, `lesson.hours.assessment`, `lesson.hours.raw`.
- Пиши для учителя-неэксперта: простым языком, с готовыми фразами и действиями.
- Критерии проверки формулируй словами, без rc/stderr/stdout в МР.
- Методическое основание пиши словами, без локаторов и ссылок на прежние табличные источники.
- Ключи и пояснения размещай здесь и в QA, не в ученическом файле практики.

### Expected outputs
- МР: цель и задачи, методическая опора, подготовка, сценарий, ключи и пояснения, типичные ошибки и реакция.

### Quality gate
- Все обязательные секции из сохранённой структуры требований присутствуют или явно отмечены как неприменимые.
- Все ссылки на источники ведут на Markdown-файлы из JSON, а не на исходные `.docx`, `.pdf`, `.html`.
- Объём, тип задач, L1/L2 и ограничения L3 соответствуют JSON.
- Сценарий, тайминг, подготовка и методические пояснения достаточны для `lesson.hours.theory`, `lesson.hours.practice`, `lesson.hours.assessment` и `lesson.hours.raw` соответствующего материала.
- Ключи и пояснения размещены только в разрешённых артефактах.
- Формулировки понятны целевой аудитории и учителю-неэксперту.

## Сохранённая структура требований
### МЕТОДИЧЕСКИЕ УКАЗАНИЯ УЧИТЕЛЮ (МР) — ТЗ

МР — для теории и практики (на самостоятельную не делается). Структура артефакта — в «Общее» → раздел 6; здесь — как его писать.

- **1. Главный принцип**: МР даёт учителю то, чего НЕТ в ученическом материале: как объяснять, что предусмотреть, как реагировать на ошибки. Рассчитан на учителя-неэксперта. Не пересказ ученического материала.
### 2. Обязательная структура

- **Цель и задачи**: Обобщающее предложение (общая цель) + нумерованный список задач. Методическим языком.
- **Методическая опора**: Что важно донести, на чём акцент.
- **Подготовка**: Среда/материалы понятным языком; что проверить заранее.
- **Сценарий**: Последовательность; тайминг; готовые фразы для учеников.
- **Ключи и пояснения**: Правильные ответы + «почему» (только здесь и в QA).
- **Типичные ошибки и реакция**: Что путают и как реагировать.
- **3. Как писать**: Простым языком, без dev-жаргона. Критерий проверки — словами. Основание — текстом, без локаторов. Готовые фразы для учеников (в т.ч. снимающие страх ошибки).
- **4. Требования**: Не дублировать ученический материал; учитель-неэксперт должен провести занятие по МР; ключи только здесь и в QA.
- **Плохо**: Хорошо
- **«Критерий: rc=0, stderr пуст, stdout совпадает»; «Основание: локатор прежнего табличного источника».**: «Программа запускается без ошибок и выводит ровно ожидаемый текст»; основание изложено словами.

## Controlled advanced additions from Python advanced v16

Source: `Python advanced prompt workbook v16`, sheet `Teacher guidance`.

- Teacher guidance must give what is absent from learner materials: how to explain, what to prepare, what to emphasize, and how to react to typical mistakes.
- It is for a non-expert teacher and must not merely retell the learner artifact.
- Required structure remains: goals/tasks, methodical support, preparation, lesson scenario/timing, typical learner mistakes, teacher responses, and keys/checking notes only where the channel permits them.
- For advanced practice, explain how the teacher should handle L3 independent-solution tasks and source-required AI-use disclosure without turning the guidance into a public solution bank.
